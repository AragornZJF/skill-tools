# Wrapup(收尾 · 发布前两件事)

发布 = 别人要继承你的未知。**固定顺序 ⑩ → ⑪**:
- **⑩ 推介文档**:对外——替评审省一遍挖未知的过程,拿 buy-in / 批准 / 合并
- **⑪ 改动测验**:对内——证明你自己真懂了这次改动再合并

顺序不反:先对外拿 buy-in,再对内自检。

## 确认门

两招都产 HTML,属产出文件类——**先确认再执行**(说清要出什么,用户点头再写文件)。

## HTML 前置(⑩⑪ 必做)

产出 HTML 前**必须先拉 `design-html` 技能**,读它的 `design-standard.md`(WHY)和 `skeleton.md`(WHAT to copy)。两条铁律:React+Babel 固定版本+integrity hash;避开 AI slop 黑名单。

---

## ⑩ 推介文档 The buy-in doc
**何时用**:写完了,要拿批准 / 合并。评审和你当初有一样的未知。
**产出**:**HTML** 单文件(`artifacts/unknowns/pitch-<slug>.html`)——
1. 开头放**动画 demo / GIF**,让评审 3 秒看懂这次改了什么
2. 需求规格
3. **`implementation-notes.md` 里的关键决策**(预答评审会担心的失败点)——直接引用 `notes` 的产出
4. 明确点名:谁需要在哪件事上签字
**执行前必读**:`design-html` 技能(设计准则、Tweaks、内容准则、交付自检流程)。
**Prompt 模板**:
```
把以下内容打包成一份 HTML,我能直接丢到群里拿批准:
1. 开头放动画 demo / GIF,让评审 3 秒看懂这次改了什么
2. 需求规格
3. implementation-notes.md 里的关键决策(预答评审会担心的失败点)
4. 明确点名:谁需要在哪件事上签字
```
**价值**:证明你已考虑到他们担心的失败点,审批自然快。

## ⑪ 改动测验 Quiz me before I merge
**何时用**:长时间编码后合并前。读 diff 只能懂皮毛,大量行为藏在既有路径里。
**产出**:**HTML** 单文件(`artifacts/unknowns/quiz-<slug>.html`)——改动上下文 + 直觉解释 + 具体改了什么 + 底部一套测验题。**必须全对才合并**;答错直接指回你略过的章节。
**执行前必读**:`design-html` 技能(设计准则、AI slop 黑名单、交互/测验的 React 模式)。
**Prompt 模板**:
```
我要确认自己完全理解了这次改动。
给我一份 HTML 报告:
1. 改动的上下文和背景
2. 直觉解释(为什么这么改)
3. 具体做了什么
4. 底部一套测验题,我必须全对才能合并;答错指回我略过的章节
```
**Thariq 的规矩**:测验全对,才许合并。

---

## 衔接(跑完 ⑩→⑪ 后)

- ⑩⑪ 都过 → 合并 / 发布。闭环完成
- 若 ⑪ 测验答错 → 指回略过的章节,补上理解再来
- 新任务 / 新 idea → 直接触发 `research` 重进知识层挖未知

## 跑完必做

- 产出匹配用户语言(中文则 HTML 文案也用中文)
- HTML 自包含:单文件、内联 CSS、不引外部资源(CDN 例外)
- ⑩ 必须引用 `implementation-notes.md` 的实际决策,不得编造
- ⑪ 测验答错必须指回具体章节,不得只判对错
- **⑩ 必在 ⑪ 前**(先对外拿 buy-in,再对内自检)
- **主动提议下一步**,形成闭环
