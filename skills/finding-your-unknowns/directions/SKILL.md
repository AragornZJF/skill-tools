# Directions(倒转域 · AI 先出方案)

解决 **未知已知(看到才知道)**——用户**看到才知道**要什么(审美、布局、手感、说不清但能认)。agent 角色 = **倒转**:不问用户,先出方案让用户判断。

> 本域装**核心技法(倒转)**。答案在人脑子里、写不进 prompt,所以 AI → 人:先甩方案/盲区,人看了再说。

## 核心动作

**不要反问用户"你想要什么"**——他说不清才进这域。直接出 2-4 个**截然不同**的方案/方向/原型,让用户指着说"这个,不要那个"。用户的判断比他的描述准。

## 确认门

- **③④ 产 HTML** → 属产出文件类,**先确认再执行**(说清要出几个方向/什么 mock,用户点头再写文件)
- **⑦ 产 MD** → 属产出文件类,**先确认再执行**

## HTML 前置(③④ 必做)

产出 HTML 前**必须先拉 `design-html` 技能**,读它的 `design-standard.md`(WHY)和 `skeleton.md`(WHAT to copy)。两条铁律:React+Babel 固定版本+integrity hash;避开 AI slop 黑名单。

## 通用第 0 步(⑦ 必须做)

⑦ 会"凭感觉编"。跑之前**必须先用 `Glob`/`Grep`/`Read` 扫描目标模块/参考源码的真实代码**,每条发现挂到具体文件/函数,不得凭通用知识编造。

## ③ 四种设计方向 Design directions
**何时用**:审美/布局类"看到才知道要什么"的需求。
**产出**:**HTML** 单文件(`artifacts/unknowns/design-directions-<slug>.html`,起步用 `design-html` 技能的 skeleton.md)——同一需求渲染成 **4 种风格迥异**的方向(如控制台/编辑/看板/终端),每个带"采纳/跳过"按钮,点击后自动拼成你的回复。
**执行前必读**:`design-html` 技能(设计准则、3+ 变体探索流程、Tweaks、AI slop 黑名单)。
**Prompt 模板**:
```
我想要[一个数据看板],但我毫无审美、也不知道技术上能做到什么程度。
做一个 HTML 页面,放 4 个风格完全不同的设计方向,
让我看了能告诉你喜欢哪个、不喜欢哪个,
并用"采纳/跳过"按钮把我的选择写进回复里。
```

## ④ 一次性原型 Mock before you wire
**何时用**:动真代码前,想先看到布局/交互。用假数据。
**产出**:**HTML** 单文件(`artifacts/unknowns/mock-<slug>.html`)——可点击的抛弃式 mock(可含 A/B 布局切换、占位数据),明确标注"这是 mock,不接真实逻辑"。
**执行前必读**:`design-html` 技能(React+Babel 固定版本、样式命名陷阱、匹配既有视觉词汇)。
**Prompt 模板**:
```
在动任何真实代码之前,先做一个独立的 HTML mock:
[新功能/工具栏]长什么样,数据用假的就行。
我想先看到布局再决定要不要往真实 app 里接。
```
**价值**:未知已知在原型期发现≈零成本;拖到实施期,一个小改动可能让代码天翻地覆。

## ⑦ 参考语义图 Point at a reference
**何时用**:说不清,但有喜欢的实现/库/网站。最好的参考是源代码(哪怕另一种语言)。
**产出**:MD 语义对照表——原文摘录 ↔ 目标实现要点、gotcha 注记、边界情况表。证明它读懂了再动手。
**Prompt 模板**:
```
[路径/URL] 这个 [语言] 实现的 [行为],正是我想要的。
先 Read 它的源码,出一份语义对照表(摘录↔要点、坑、边界情况),
确认你理解后再用同样语义在咱们的 [目标语言/模块] 重实现。
```

## 衔接(跑完一招后主动提议)

- 倒转域跑完(用户已指着选了方向)→ 汇到 `plan`(⑧ 可调计划)进实施
- 若 ⑦ 跑完发现还需要挖"已知未知"(能问的)→ 交 `research`
- ③④ 的 HTML 产出**不要接真实逻辑**——它是 mock/方向,确认方向后再用 `plan` 规划真接入

## 跑完必做

- 产出匹配用户语言(中文则 HTML 文案也用中文)
- HTML 自包含:单文件、内联 CSS、不引外部资源(CDN 例外)
- 占位优于拙劣的真尝试:没有素材就画占位,别编
- **主动提议下一步**,形成闭环
