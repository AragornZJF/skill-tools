# Phase 1 — 交互式需求定义（Brainstorming）

在编写任何代码之前，先把模糊想法细化成 `specs/<topic>.md`。

**硬性门禁：** 在完成整个 brainstorming 流程并获得用户批准之前，不得调用任何实现技能、写任何代码、搭建任何项目骨架。

## 核心原则

- **一次一个问题** — 不要同时抛出多个问题
- **优先选择题** — 比开放题更容易回答
- **YAGNI 无情** — 从所有设计中砍掉不必要的功能
- **探索替代方案** — 在确定之前，始终提出 2-3 种方案
- **增量确认** — 分块呈现设计，每块确认后再继续
- **保持灵活** — 当某部分不清楚时，回头澄清

## 检查清单

按顺序完成以下每项：

### 1. 探索项目上下文

检查当前项目的文件、文档、最近的提交记录，了解已有代码和架构。

### 2. 提出澄清问题

**必须使用 `question` 工具**（终端选择菜单，自动附带自定义输入兜底），一次一个问题，不可同时抛出多个问题。

依次用 `question` 工具提问：
- **目的（Purpose）** — 要解决什么问题（选项示例：`["新功能", "Bug 修复", "重构", "性能优化"]`）
- **约束（Constraints）** — 技术/时间/团队限制（选项示例：`["Node.js", "Python", "无特定技术限制", "有 deadline"]`）
- **成功标准（Success Criteria）** — 怎样算做好（选项示例：`["通过现有测试", "新增 E2E 测试覆盖", "满足验收标准"]`）

如果需求描述包含多个独立的子系统（如"做一个带聊天、文件存储、计费和分析的平台"），立即指出需求需要先分解，而不是继续细化细节。

对于范围过大的项目，帮助用户分解为子项目：独立的模块是什么、如何关联、按什么顺序构建。然后对第一个子项目走完整 brainstorming 流程。每个子项目独立走 spec → plan → 实现循环。

### 3. 提出 2-3 种方案

- 给出 2-3 种不同方案，附带各自的权衡
- **使用 `question` 工具**在终端展示方案选项，每项 label 为方案简称、description 为权衡说明，首选项标注 `(Recommended)`
- 推荐你的首选方案并说明理由

### 4. 分块呈现设计

- 确认理解后，分块呈现设计
- 每块的内容量根据复杂度调整：简单的几句话，复杂的 200-300 字
- **每块讲完后使用 `question` 工具**询问"这部分看起来对吗？"，选项：`["正确，继续", "需要修改"]`
- 覆盖：架构、组件、数据流、错误处理、测试策略
- 把系统拆成职责单一的小单元，每个单元可通过明确定义的接口通信，可独立理解和测试

**在已有代码库中工作：**
- 先探索现有结构，遵循已有模式
- 如果现有代码影响当前工作（如文件过长、边界不清），将针对性改进纳入设计
- 不提出无关的重构

### 5. 编写 specs

将确认后的设计写入 `specs/<topic>.md`（Ralph 格式）：

```
# <主题名称>

## JTBD（Jobs To Be Done — 用户雇佣此功能要完成的任务）
<此主题服务于什么用户需求>

## 范围（Scope）
<一句话描述，不含"和">

## 验收标准（Acceptance Criteria）
<可观察、可验证的结果>

## 约束（Constraints）
<技术/业务限制条件>
```

### 6. Spec 自检

写完 spec 后做以下检查，发现问题立即修复：

1. **占位符扫描** — 是否有"TBD"、"TODO"、不完整的段落、模糊的需求？
2. **内部一致性** — 各部分是否有矛盾？架构是否匹配功能描述？
3. **范围检查** — 是否聚焦在单个实现计划内，还是需要进一步分解？
4. **歧义检查** — 是否有需求可以被理解为两种不同的方式？如果有，选一个并明确表达。

### 7. 用户审查 spec 文件

自检通过后，请用户审查写好的 spec 文件：

> "Spec 已编写并提交到 `<路径>`。请审阅，如需修改请告诉我，然后我们再进入实现计划阶段。"

等待用户反馈。如果用户要求修改，修改后重新运行 spec 自检。用户批准后才进入下一步。

### 8. 过渡到 Phase 2

自动执行首次运行设置（创建 `loop.sh`、`clean-state.sh`、`AGENTS.md`——见 SKILL.md 的"首次运行设置"章节），然后调用 `phases/writing-plans.md` 产出 `IMPLEMENTATION_PLAN.md`。

> "Phase 1 完成。首次运行设置已自动完成。接下来使用 writing-plans 细化实现计划。"

## 反模式："这个太简单，不需要设计"

每个项目都要走这个流程。Todo 列表、单功能工具、配置变更——全部要走。越是"简单"的项目，未检查的假设造成的浪费越大。设计可以很短（真正简单的项目几句话就够了），但你必须呈现并获批准。
